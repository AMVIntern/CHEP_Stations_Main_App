## 1) Core Architecture Overview

### Core project (VisionApp.Core)
Contains NO HALCON/PLC/UI logic.

Key domain types:
- TriggerKey(cameraId, index)
- TriggerEvent(key, timestamp)
- CaptureRequest(cycleId, triggerKey)
- FrameArrived(cycleId, triggerKey, HImage image, capturedAt) : IDisposable
- InspectionResult(cycleId, triggerKey, pass, message)
- CycleCompleted(cycleId, overallPass, results[], completedAt)

Key services/interfaces:
- ITriggerSource -> IAsyncEnumerable<TriggerEvent> ReadAllAsync(CancellationToken)
- ICamera -> Task<FrameArrived> CaptureAsync(CaptureRequest, CancellationToken)
- IInspectionRunner -> runs inspection per frame (dummy for now)
- IImageLogger -> LogFrameAsync(FrameArrived, ct)
- IResultSink -> write final results somewhere (null for now)
- IFrameObserver -> UI observer for frames arriving

Engine:
- CapturePlan:
  - OrderedTriggers: list of TriggerKey (for validation/expected count)
  - StartTriggers: HashSet<TriggerKey>
  - EndTriggers: HashSet<TriggerKey> (multi-end supported)
  - ExpectedCount = OrderedTriggers.Count
- CycleEngine:
  - Deterministic state machine:
    - Idle -> Running -> Completing
  - On TriggerEvent:
    - if Idle and trigger in StartTriggers => StartNewCycle
    - ignores unknown/unplanned triggers
    - prevents duplicates per cycle using _seenTriggers
    - marks completion-ready only when ALL EndTriggers have been seen (multi-camera end)
  - On InspectionResult:
    - cycle completes when:
      - end condition satisfied (end triggers seen)
      - AND result count == ExpectedCount
  - Important fix implemented:
    - if start trigger is also end (single trigger cycle) => immediately move to Completing

Hosted pipeline services:
- TriggerDispatchService: reads triggers and writes to channels
- CameraDispatcherService: receives CaptureRequest, calls ICamera, produces FrameArrived
- InspectionService: receives FrameArrived, calls IInspectionRunner, produces InspectionResult
- CycleOrchestrationService: applies CycleEngine logic and emits CaptureRequests + CycleCompleted
- FrameObserver wiring sends frames to CycleFramesStore (UI + tiles)

## 6) HALCON camera class: HalconGigECamera

HalconGigECamera implements ICamera.
Responsibilities:
- EnsureConnectedAsync() with reconnect/backoff
- DrainQueue(handle) to drop stale buffered images
- TryGrabAsync(handle) using GrabImageAsync (single retry on HALCON 5322)
- CaptureAsync(request) returns FrameArrived with HImage

Important: Simplified HalconGigECamera.cs to remove:
- ExposureTime, ExtraParams
- GrabTimeoutMs / RetryOn5322 options
- ReconnectBackoff options
Instead uses internal constants:
- GrabMaxDelayMs = 0
- RetryDelayMs = 5
- backoff start 250ms, max 2000ms

Health:
- HalconCameraHealthService (BackgroundService) checks cameras periodically and calls EnsureConnectedAsync.

## 8) PLC Triggers (libplctag) - Refactor

Old app had hardcoded:
- PlcIp, CipPath, BaseTag for each camera group
- fixed tag count arrays size=5
- sync gating: ignore triggers 2..N until trigger1 seen
- re-arm requires low sustained MinLowMs
- poll timer PeriodicTimer every ReadDelayMs (e.g., 5ms)
- per-tag reads done individually

New app uses config-based groups:

PlcTriggerOptions:
- Gateway, Path, ReadDelayMs, MinLowMs
- Groups: list of PlcTriggerGroup
PlcTriggerGroup:
- CameraId (or CameraIds in newer version)
- BaseTag (e.g., CameraStation4_1_Control.AppTriggerIndex)
- TriggerCount (e.g., 4)
- RequireSyncOnTrigger1 (true)

Design:
- PlcTriggerSource.ReadAllAsync yields TriggerEvent stream
- GroupMonitor per group monitors BaseTag.1..BaseTag.N
  - Tag[] _tags
  - int[] _prev
  - bool[] _armed
  - long[] _lastLowTicks
  - bool _synced

Issue fixed:
- CS1626 yield return in try/catch: solution is move try/catch inside loops, not around the yield.

Config examples:
Single camera per group:
"Groups": [
  { "CameraIds": ["Cam1"], "BaseTag": "..._1_...AppTriggerIndex", "TriggerCount": 4, "RequireSyncOnTrigger1": true },
  { "CameraIds": ["Cam2"], "BaseTag": "..._2_...AppTriggerIndex", "TriggerCount": 4, "RequireSyncOnTrigger1": true }
]

If one BaseTag drives multiple cameras (fan-out), use:
{ "CameraIds": ["Cam2","Cam3"], "BaseTag": "CameraStation4_2_Control.AppTriggerIndex", ... }

NOTE: must use CameraIds (plural), not CameraId, if group supports many cameras.

## 10) Known Runtime Issue: Missing First Trigger / Wrong First Image

Symptom:
- Always missing first trigger image.
- First displayed image corresponds to real “second image”.
- Sometimes PLC feels slow / misses edges.

Hypotheses:
A) PLC Trigger1 edge missed due to polling too slow or pulse too short
B) Trigger1 detected but ignored due to sync/armed gating (RequireSyncOnTrigger1 + MinLowMs)
C) Trigger1 detected, but camera returns buffered next frame because DrainQueue discards the correct first frame

Important reasoning:
- PLC poll target = ReadDelayMs (e.g., 5ms), but REAL tick can be slower due to multiple tag.Read() calls
- If PLC pulse high time < poll tick time, rising edge may be missed
- DrainQueue before GrabImageAsync might discard Trigger1 frame if it was already buffered

Planned solution step:
Add PLC trigger logging diagnostics first before changing logic.

- UI observer + image logger must skip uninitialized images

## Core Pipeline Architecture (Hosted Services + Channels)

### Channels used

Originally:

* `Channel<TriggerEvent>`
* `Channel<CaptureRequest>`
* `Channel<FrameArrived>`
* `Channel<InspectionResult>`

Later changed to support preprocessing:

* `Channel<RawFrameArrived>` (raw from camera)
* `Channel<FrameArrived>` (processed frame used by UI/logging/inspection)

Also changed from **Unbounded** to **Bounded** for raw + processed frames to control memory:

* `RawFrameArrived` bounded capacity ~30 (DropOldest)
* `FrameArrived` bounded capacity ~30 (DropOldest)

### Hosted services (Core.Hosting.Services)

1. **TriggerPumpService**

* `await foreach (var t in triggerSource.ReadAllAsync(ct))`
* writes to trigger channel
* This behaves like “infinite stream” until cancellation — it awaits triggers.

2. **CycleEngineService**

* reads triggers channel + inspection results channel
* single deterministic state machine `CycleEngine`
* uses a `SemaphoreSlim gate` to serialize access to CycleEngine (since two read loops)
* emits `CaptureRequest` into capture channel

3. **CameraDispatcherService**

* reads `CaptureRequest`
* calls `ICamera.CaptureAsync`
* writes **RawFrameArrived** into raw frame channel (after refactor)
* Previously it also notified UI + logging directly, but after preprocessing split those moved to preprocess service.

4. **FramePreprocessService** (added)

* reads `RawFrameArrived`
* runs `IFramePreprocessor` (Gamma preprocessor)
* **disposes raw image if a new processed frame was created**
* notifies UI observers with processed frame
* logs processed frame (what you inspected)
* writes processed `FrameArrived` into processed frame channel

5. **InspectionService**

* reads processed `FrameArrived`
* runs `IInspectionRunner`
* writes `InspectionResult`
* always `frame.Dispose()` in finally to free HALCON resources

## Cycle Engine Logic

### CapturePlan

* `OrderedTriggers` list (expected count + readability)
* `StartTriggers` set
* `EndTriggers` set (multi-end supported)
* `ExpectedCount = OrderedTriggers.Count`

Example plan:

* Cam1 triggers 1..4
* Cam2 triggers 1..4
* Start: Cam1[1], Cam2[1]
* End: Cam1[4], Cam2[4]

### CycleEngine states

* `Idle`
* `Running`
* `Completing`

Key behaviors:

* In `Idle`, only starts new cycle if trigger is in StartTriggers and is contained in plan.
* In Running/Completing:

  * ignores triggers not in plan
  * dedupes using `_seenTriggers`
  * tracks end triggers seen with `_endSeen`
  * moves to Completing only when **all end triggers** seen
* Completes only when:

  * all end triggers seen
  * AND results count equals ExpectedCount
* Special fix: if start trigger is also end trigger (single trigger cycle), immediately Completing.

## PLC Trigger System (libplctag)

### PlcTriggerSource

* Builds multiple `GroupMonitor` from config
* Uses PeriodicTimer at `ReadDelayMs` (e.g. 5ms)
* Polls tags per group per tick
* `GroupMonitor` monitors BaseTag.1..BaseTag.N

### GroupMonitor logic

Per trigger index i:

* Tag reads bit0 (0/1)
* `_armed[i]` becomes true only after sustained LOW for `MinLowMs`
* Rising edge detected if `_prev=0`, `value=1`, and `_armed=true`
* Sync gating: if `RequireSyncOnTrigger1=true`, ignore edges for triggers 2..N until trigger1 is seen once after startup
* Fan-out support: one PLC tag group can produce trigger events for multiple camera IDs

### Config issues found/fixed

* Needed correct section naming / binding alignment.
* Logging section initially had mismatch `PlcTriggers` vs `PlcTrigger` in config.
* Ultimately: created `PlcTriggerOptions` with diagnostics flags.

### Missing-first-trigger investigation

We had a known issue where first image/trigger seemed shifted.
Possible causes:

* PLC pulse too short relative to actual polling tick time
* gating logic causing ignoring of trigger1
* camera buffer returning wrong frames due to DrainQueue behavior
  Eventually user discovered they had inverted PLC timing tags per camera; after correction it behaved.

## HALCON Camera Capture (HalconGigECamera)

* Uses HDev procedures for open/close so all camera param setup lives inside HALCON (TriggerMode, StreamBufferHandlingMode, AcquisitionMode=Continuous, etc.)
* Connection logic with retry/backoff, with optional global open/close gate to avoid driver weirdness
* Capture uses:

  * EnsureConnectedAsync
  * Grab lock to serialize GrabImageAsync

* If grab fails, returns **uninitialized `new HImage()`** to avoid stalling pipeline (then preprocessing/logging/inspection skip uninitialized images).

**Key decision:** keep DrainQueue because camera stays “armed” and buffers frames; when triggered we want newest frame.

You wanted UI to display **the same image you process**, and you didn’t want to apply gamma twice.

## Disposal / Memory Ownership Rules (Very Important)

HALCON images are heavy, must be disposed correctly.

Ownership rules:

* **Pipeline owns** raw/processed frame images until they are handed to UI as a clone.
* UI must **CopyImage()** and own that copy.

Where to dispose:

* **FramePreprocessService**

  * If preprocessor returns a *new* `FrameArrived`, dispose the raw frame (`raw.Dispose()`).
  * Do not dispose processed frame here (it continues down pipeline).
* **InspectionService**

  * Always dispose `FrameArrived` after inspection (in finally).
* **UI store / tiles**

  * UI observer does `frame.Image.CopyImage()`
  * Tile VM disposes old image before replacing
  * Store clears tiles on new cycle and disposes old images
* **Image logging**

  * Logger clones image and enqueues clone
  * If queue drops item, must dispose clone immediately
  * Writer must dispose clone after saving

## Image Logging (Root cause of memory growth)

Observation:

* Offline mode stable RAM.
* Online mode RAM climbed rapidly until ImageLogging disabled.
* Root cause was: logging pipeline could not keep up → unbounded growth (or clones not disposed on drops).

Fixes:

* Use **bounded channels** for raw/processed frames (capacity 30, DropOldest)
* Also ensure logging queue is bounded and drop behavior disposes images.

Key learning:

* A channel gets “full” when producer is faster than consumer. If consumer (disk writes) can’t keep up, items pile up.
* With large objects (HALCON images), unbounded queues == memory explosion.

Settings guidance for ImageLogging (example):

* `QueueCapacity`: start modest (e.g., 100–300); if writing is slower than capture, you MUST drop or block.
* `DropWhenBusy=true` recommended for live systems so logging never kills app.
* `Format=png` is heavier; jpg is faster/smaller.
* Folder structure options OK (month/day/cycle).

### Symptom

### Steps

### Key learning about bounded channels

* A bounded channel fills if producers write faster than consumer reads.
* If FullMode=DropOldest or DropWrite, items can be dropped.
* If dropped items are IDisposable, dropping without disposal can leak unless handled carefully.
* If DropWhenBusy=false (Wait), pipeline can slow drastically due to backpressure.

## 10) Image Logging Issue: Missing Images vs Performance

### Observed behavior

* When setting `DropWhenBusy=false`, app becomes slow and UI goes out of sync.
* When `DropWhenBusy=true`, app stays real-time but images are sometimes missing “randomly.”

### Explanation

* With `DropWhenBusy=false`, enqueue waits when log queue full → blocks preprocess loop → stalls pipeline → UI delayed/out of sync.
* With `DropWhenBusy=true`, queue uses DropOldest → when writer can’t keep up, log items are dropped silently → missing images.
* Old app used “fire-and-forget threads” (high concurrency) which increased throughput but risks CPU/disk thrash + unbounded memory.

### Recommended tradeoff

* Real-time priority: keep DropWhenBusy=true and optimize logging throughput:

  * prefer JPEG over PNG
  * increase QueueCapacity
  * potentially controlled multi-writer design (not unlimited fire-and-forget)
* Completeness priority: DropWhenBusy=false but accept pipeline slowdown.

# Defect Assignment: What it is and how it fits into the app

## The goal

After YOLOX (and any other inspections) run per-frame, we want to **post-process the detections** into a **station-level summary** that:

1. **Assigns each detected defect to a physical product element** (e.g., TLB1, TIB1…)
2. Optionally assigns to a **bearer element** (B1/B2/B3) based on trigger index
3. Applies **station-specific rename rules** (e.g., PN → TN on certain triggers)
4. Produces a final **dictionary of counts** that is easy to:

   * log to CSV in a fixed column format
   * drive downstream actions (future: tower light, PLC output, etc.)

This is *not* per-image UI overlay logic. It’s **station-level “business logic”**, built as a post-processing pipeline triggered by inspection results.

---

# Where Defect Assignment sits in the runtime flow

## The full flow (high level)

1. **Trigger + Capture**

   * PLC (or Dummy) produces TriggerEvents
   * CycleEngine produces CaptureRequests
   * Cameras capture images -> `FrameArrived` (CycleId + TriggerKey + HImage)

2. **Inspection**

   * `InspectionService` reads `FrameArrived` from the channel
   * It calls `IInspectionRunner.InspectAsync(frame)`
   * The runner executes steps in order (YOLOX, filter, decision, etc.)
   * Runner returns an `InspectionResult` (Pass/Score/Message + optional Visuals/Metrics)

3. **Observers**

   * Inspection results are published to any `IInspectionObserver`
   * We added “Defect Assignment + Station Aggregation” observers here.
   * This is the cleanest professional point to do “results post-processing”.

Defect assignment does **not** run inside YOLOXStep itself. It runs after inspection results exist.

### Your requirement

You said:

That’s why we introduced:

4) PLC PALLETID READ-AT-TRIGGER, WRITE-AT-COMPLETION
------

Overview:
The PLC integration has been updated to read a PalletID from the PLC control tag at the first trigger for each station, then write that same PalletID back to the results tag after all defect data. This replaces the previous HHmmss timestamp write and implements a cleaner handshake protocol.

Flow:
1. Trigger fires (index 1) for a station → PlcTriggerSource.GroupMonitor detects rising edge
2. Reads DINT PalletID from configured control tag (e.g., "sideStation_1_Control.PalletID")
3. Stores in IPalletIdStore keyed by station (e.g., "Station4")
4. Cycle completes → PlcStationDefectBoolSink writes all defect bools
5. Writes the stored PalletID DINT to results tag (e.g., "results_sideStation.PalletID")

Configuration (appsettings.json):
Location: VisionApp.Wpf/appsettings.json, PlcTriggers.Groups section
Each trigger group now has:
  "PalletIdControlTag": "sideStation_1_Control.PalletID"
  "PalletIdControlTag": "sideStation_2_Control.PalletID"
  "PalletIdControlTag": "bottomLeadingStation_Control.PalletID"

PalletID Store:
Location: VisionApp.Infrastructure/PlcOutbound/
New files:
  - IPalletIdStore.cs: Interface with Set(stationKey, palletId) and Get(stationKey)
  - PalletIdStore.cs: ConcurrentDictionary<string, int> implementation (thread-safe, singleton)

Trigger Source Updates:
Location: VisionApp.Infrastructure/Triggers/PlcTriggerSource.cs
Changes:
  - Constructor now injects IPalletIdStore and ICameraStationResolver
  - GroupMonitor constructor receives these deps and resolves station key from first camera
  - InitializeTags creates a DINT Tag object for PalletIdControlTag (if configured)
  - PollOnce: when trigIndex == 1 fires, reads DINT synchronously and calls store.Set(stationKey, value)
  - On read failure, stores 0 (graceful degradation)
  - Dispose cleans up PalletID tag

Defect Sink Updates:
Location: VisionApp.Infrastructure/PlcOutbound/PlcStationDefectBoolSink.cs
Changes:
  - Constructor now injects IPalletIdStore
  - After enqueuing all defect bools, instead of computing HHmmss timestamp:
    - Calls _palletIdStore.Get(row.StationKey) to retrieve the read value
    - Enqueues PlcDintWrite with that value
  - Tag name and config key (TimestampTagSuffix) remain the same
  - Only the VALUE written has changed (timestamp → PalletID)

DI Registration:
Location: VisionApp.Infrastructure/DI/VisionInfrastructureModules.cs
In AddPlcOutboundModule:
  - IPalletIdStore registered as singleton (PalletIdStore implementation)

Threading & Safety:
  - PalletID.Set called from polling thread (PlcTriggerSource)
  - PalletID.Get called from write queue consumer thread (PlcStationDefectBoolSink)
  - ConcurrentDictionary ensures thread safety without additional locks
  - DINT read is synchronous (happens once per cycle per station at trigger index 1)
  - Cycles are sequential; no risk of next-cycle overwrite during current write

Known Behavior:
  - If PalletIdControlTag is empty/null in config, the feature is disabled for that group (graceful)
  - If PLC read fails, 0 is written to results (logged as warning)
  - If both S4 trigger groups have PalletIdControlTag, first to trigger index 1 wins (usually not an issue since they're the same pallet)

6) PRODUCTION COUNTER PERSISTENCE ACROSS RESTARTS
------

Overview:
The shift-based production counters (Total, Assured, Standard) now persist to disk, allowing them to survive application restarts within the same shift. If the app is restarted and the shift has changed, counters reset to 0 for the new shift.

File Format & Location:
Path: C:\ProgramData\AMV\VisionApp\0.0.1\Counts\production_counts.json
Example content:
{
  "ShiftNumber": 2,
  "ShiftDate": "2026-04-01",
  "Total": 150,
  "Assured": 140,
  "Standard": 10
}

Persistence Logic:
Location: VisionApp.Wpf/Services/ShiftProductionCounterSink.cs
Changes:
  - Added const paths for CountsDir and CountsFilePath
  - Added nested CountsSnapshot DTO for JSON serialization
  - Constructor now:
    - Resolves current shift via _shiftResolver.Resolve(DateTimeOffset.Now)
    - Pre-sets _lastShift = currentShift (prevents spurious reset on first cycle)
    - Calls TryRestoreFromFile(currentShift)
  - UpdateMetrics now calls SaveToFile() after each counter increment
  - Injects ILogger<ShiftProductionCounterSink> for error logging

Load on Startup (TryRestoreFromFile):
  1. Checks if production_counts.json exists
  2. Reads and deserializes the JSON file
  3. Compares saved (ShiftNumber + ShiftDate) with current shift
  4. If both match → restores Total, Assured, Standard into ProductionCounterStore
  5. If shift differs → file is stale; start at 0
  6. Logs informational message about restore success or reason for skip
  7. Graceful error handling: file errors logged as warnings, never crash the app

Save After Each Result (SaveToFile):
  1. Creates CountsDir if it doesn't exist (idempotent Directory.CreateDirectory)
  2. Serializes current state (ShiftNumber, ShiftDate, Total, Assured, Standard) to JSON
  3. Overwrites the same file with new values (single file per installation)
  4. Writes are synchronous (called from dispatcher thread, file is ~130 bytes)
  5. Write exceptions are caught and logged as warnings (non-blocking)

Shift Boundary Behavior:
When UpdateMetrics detects a shift change:
  1. Resets counters to 0
  2. Updates _lastShift to the new shift
  3. UpdateMetrics increments from the fresh state
  4. SaveToFile writes the first result of the new shift (typically Total=1, Assured/Standard depends on pass/fail)

Restoration Scenarios:
| Scenario | Result |
|----------|--------|
| App restart, same shift (same date + number) | Counters restored from file |
| App restart, new shift (different date or number) | Counters reset to 0 for new shift |
| File deleted manually | App logs info, starts at 0, creates new file on first cycle |
| File corrupted or unreadable | App logs warning, starts at 0 gracefully |
| File permissions denied | App logs warning, starts at 0; next cycle creates new file |

Integration Points:
  - No UI changes (uses existing ProductionCounterStore bindings)
  - No DI changes (ShiftProductionCounterSink already a singleton)
  - Config: No new config section needed (simple file-based persistence)
  - Logging: Uses ILogger<ShiftProductionCounterSink> for all file operation messages

Known Behavior:
  - File is small (~130 bytes) and safe for frequent writes (every cycle)
  - ShiftDate format: "yyyy-MM-dd" (ISO format, locale-independent)
  - If app crashes mid-write, next startup will either use old values (if write incomplete) or new values (if write completed)
  - Field teams can manually delete the counts file to reset; no special cleanup needed
